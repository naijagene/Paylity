# VTPass Request ID Policy

## Format

```
{PAYLITY_REFERENCE}-F{attempt_number}
```

Example: `PYL-20260710-ABC123-F01`

- `F` prefix denotes fulfillment attempt
- Attempt number zero-padded to 2 digits (01–99)
- Generated by `VTPassRequestIdGenerator::forAttempt()`

## Rules

| Scenario | Request ID |
|----------|------------|
| First fulfillment attempt | New ID (`-F01`) |
| Retry after confirmed failure | New ID (`-F02`, `-F03`, …) |
| Uncertain timeout | **Retain** original ID |
| Reconciliation requery | **Retain** original ID |
| Duplicate provider response | Requery original ID; never blind resubmit |

## Persistence

1. Attempt row created with `request_id` before HTTP call
2. `markSubmitted()` records payload
3. Unique DB constraint on `request_id`

## Deprecated

Checkout `request_payload.request_id` is no longer reused across attempts. Adapters build a placeholder ID; orchestrator overrides with attempt ID before submit.
